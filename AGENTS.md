# AGENTS.md

## Проект

Android anime viewer приложение на Kotlin, Jetpack Compose и Material 3.

Цель проекта — аккуратное Android-native приложение для просмотра аниме с понятной архитектурой, стабильной сборкой и возможностью постепенно подключать разные источники.

## Карта репозитория

Верхний уровень:

* `app` - Android-приложение: UI, навигация, тема, настройки, экраны
* `parsers` - общая библиотека для поиска тайтлов, метаданных и резолва видео
* `docs` - документация и UI-гайд
* `gradle` - wrapper, версии зависимостей и общая Gradle-конфигурация

Ключевые файлы:

* `settings.gradle.kts` - подключение модулей
* `build.gradle.kts` - корневая сборка
* `gradle/libs.versions.toml` - версии библиотек
* `app/build.gradle.kts` - Android-зависимости и конфиг приложения
* `parsers/build.gradle.kts` - сборка библиотеки парсеров
`app`:

* `app/src/main/java/org/akkirrai/hibiki/app` - навигация, theme, settings
* `app/src/main/java/org/akkirrai/hibiki/core` - design system, model, network, source
* `app/src/main/java/org/akkirrai/hibiki/feature` - `home`, `search`, `details`, `library`, `settings`
* `app/src/main/res` - строки, темы, цвета, drawables, mipmaps и XML-ресурсы

`parsers`:

* `core` - общий резолв и матчинг тайтлов
* `extractor` - извлечение ссылок и потоков с площадок
* `provider` - провайдеры источников
* `metadata` - Shikimori metadata source
* `validator` - проверка HLS/DASH/MP4 потоков
* `network` - HTTP и URL helpers
* `model` - общие модели парсеров
* `src/test` - тесты экстракторов, провайдеров и core-логики

## Источники

Источники подключаются через общий контракт.

Источник может предоставлять:

* поиск аниме
* детали тайтла
* список серий
* варианты просмотра
* постеры
* статусы
* метаданные

Логика конкретного источника должна быть изолирована в своём модуле или пакете.

Ошибка одного источника должна обрабатываться локально и показываться пользователю через нормальное UI-состояние.

### Предпочтения по источникам

При выборе нового источника или доработке существующего соблюдать приоритет надёжности:

1. Публичный JSON / GraphQL API без логина и без браузерной сессии
2. Публичный API с простыми header-based требованиями, если без них нельзя
3. Стабильные embed / player endpoints, если они документированы или уже устойчиво используются
4. HTML-парсинг только как последний fallback, если более надёжного варианта нет

Для metadata, поиска и списков тайтлов:

* предпочитать API-источники, а не HTML-страницы;
* не строить новую ключевую функциональность на хрупком парсинге DOM;
* если есть выбор между менее полными данными из API и более полными данными из HTML, по умолчанию выбирать API;
* если HTML всё же нужен, изолировать его в отдельном provider/extractor и считать временным или fallback-решением.

Для предложений по архитектуре и новым интеграциям агент должен:

* сначала предлагать надёжные варианты: API без логина, JSON endpoints, GraphQL, официальные или стабильные public endpoints;
* явно предупреждать, если вариант завязан на HTML, Cloudflare, cookie/session, browser emulation или нестабильную разметку;
* не предлагать без необходимости решения уровня WebView/browser automation для metadata search;
* по возможности использовать источники, которые уже возвращают внешние id (`shikimori`, `anilist`, `mal`) или другие хорошие ключи для локального матчинга.

Для поиска метаданных:

* основной search pipeline должен оставаться API-first;
* дополнительные источники можно использовать как candidate pool, confidence booster или poster fallback;
* не добавлять новый источник в основной список выдачи, если он нужен только для локализации, alias-ов или постеров;
* избегать схем, где на каждый результат поиска делается отдельный сетевой запрос, если можно сделать локальный match по уже полученным данным.

Плеер должен быть отдельным слоем.

Прогресс просмотра сохраняется отдельно.

## Правила изменений

Для каждой задачи:

1. Определи область изменения
2. Затрагивай только связанные файлы
3. Сохраняй текущую архитектуру
4. Выноси повторяемый код в общий компонент
5. После изменения проверь сборку или укажи, почему проверка не выполнена

Если задача связана с источниками или metadata search:

* сначала проверь, можно ли решить её через уже существующие API и текущие модели;
* если предлагается новый источник, кратко оцени его по надёжности: API / embed API / HTML;
* если источник опирается на HTML, отдельно объясни, почему без этого нельзя;
* при равных вариантах выбирать тот, где меньше хрупкости, меньше сетевых запросов и проще локальный match.
